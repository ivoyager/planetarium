# Spacecraft Trajectory & Ephemeris Data

Spacecraft trajectories (`tree_components/trajectory.gd`, built from `orbits.tsv` segments that are listed in `trajectories.tsv` and assigned to a craft in `spacecrafts.tsv`) are **patched-conic**: an ordered set of `IVOrbit` segments, each a conic about one gravitational primary, joined at sphere-of-influence (SOI) crossings. Heliocentric **cruise** legs are flagged `fix_gaps` and re-fitted at new-game to meet the **flyby** anchor legs (see *Gap closing* below). The data is generated by `tools/horizons_trajectory.py`.

## Authoritative source — NASA/JPL HORIZONS

All real spacecraft (and body) ephemeris data comes from the JPL Solar System Dynamics **HORIZONS** system — the same group that produces the DE planetary ephemerides. It is *the* primary source; **do not use third-party mirrors or aggregators.**

- Web app: https://ssd.jpl.nasa.gov/horizons/
- REST API (use this for the conversion pipeline): https://ssd.jpl.nasa.gov/api/horizons.api
- HORIZONS ingests the official reconstructed-trajectory **SPICE SPK kernels** from NASA **NAIF** (https://naif.jpl.nasa.gov/) — the underlying archival source and the right provenance citation. We query HORIZONS rather than raw SPK because HORIZONS performs the SPK→osculating-Keplerian-element conversion that `orbits.tsv` needs, with no SPICE-toolkit dependency.

## Element query

Request osculating elements with `EPHEM_TYPE='ELEMENTS'`, `REF_PLANE='ECLIPTIC'` (→ ecliptic of J2000, matching the `orbits.tsv` default plane), `REF_SYSTEM='ICRF'`, `OUT_UNITS='KM-S'`, `CENTER=<primary>`, `COMMAND='<id>'`, plus `START_TIME`/`STOP_TIME`/`STEP_SIZE`.

`CENTER` is the segment's gravitational primary, and **switching it is how patched-conic segments are produced** — query the *same* craft with `CENTER='@10'` (Sun, heliocentric cruise), `'@399'` (Earth, post-launch), `'@599'` (Jupiter flyby), etc., over successive time windows split at SOI crossings. Example (Voyager 1 heliocentric elements):

```
https://ssd.jpl.nasa.gov/api/horizons.api?format=text&COMMAND='-31'&EPHEM_TYPE='ELEMENTS'&CENTER='@10'&REF_PLANE='ECLIPTIC'&REF_SYSTEM='ICRF'&OUT_UNITS='KM-S'&START_TIME='1980-01-01'&STOP_TIME='1980-01-02'&STEP_SIZE='1d'
```

## HORIZONS ELEMENTS → `orbits.tsv` mapping

Verified against `program/table_orbit_builder.gd`. Write values in each column's declared unit (deg, km); the table importer converts to radians/internal units.

- `EC`→`eccentricity`, `IN`→`inclination`, `OM`→`longitude_ascending_node`, `W`→`argument_periapsis`; `parent` = the `CENTER` body (`STAR_SUN`, `PLANET_JUPITER`, …).
- **Size/phase:** use `semi_parameter` = `QR`·(1 + `EC`) [km] and `time_periapsis` from `Tp` for *every* segment — this open-conic form is valid at any eccentricity (the builder rejects `semi_major_axis`/`mean_anomaly_at_epoch`/`mean_motion` once e ≥ 1, which the hyperbolic flyby and escape legs are; the near-launch leg is mildly elliptic). HORIZONS returns `A` < 0 and `AD`/`PR` = 1e99 sentinels for hyperbolas — ignore them.
- `fix_gaps` = `x` on heliocentric **cruise** legs (`CENTER='@10'`); blank on flyby/departure anchors (see *Gap closing*).
- Leave `orbit_gravitational_parameter` blank — a segment inherits its parent body's GM.

## Gap closing, segmentation & the converter

A **flyby/departure** segment (`CENTER` ≠ `@10`) is a planet-centric osculating conic — an accurate anchor, and frame-portable (independent of where the planet is). A **cruise** segment (`CENTER='@10'`) is heliocentric and must terminate exactly at the flyby anchors' endpoints; authored from natural osculating elements it does not, leaving gaps. So cruises are flagged `fix_gaps`, and at **new-game** `IVTrajectory._fix_gaps` re-fits each flagged cruise as the conic (Lambert) through its neighbor segments' boundary positions, computed from the bodies' *actual* runtime positions. Continuity therefore survives a change of planet-orbit model (`IVRealPlanetOrbit` vs `IVOrbit`) or data source. It runs new-game only — the fixed orbits persist with the planet orbits they were fit to, so a load needs no re-fit (`fix_gaps` is transient and not persisted). Generic two-body math lives in `IVOrbit.solve_lambert` and `IVOrbit.create_from_state_vectors_and_precessions`.

**Earth-launched interplanetary craft start with a geocentric Earth departure segment** (`CENTER='@399'`, a `fix_gaps`-false anchor): it captures the launch escape hyperbola, makes the first cruise an interior two-anchor fit, and is visible when zoomed to Earth. The converter sets this segment's `segment_begin` to the conic's own `time_periapsis` — perigee is the hyperbola's nearest point to Earth (a few hundred km up, the synthetic departure), so the arc begins at the surface rather than tens of Earth-diameters up the incoming asymptote, where a bare launch-date `begin` would otherwise place it (hours before perigee). (Earth-orbiters like ISS/Hubble are plain orbits, not trajectories.)

`tools/horizons_trajectory.py` automates the per-segment HORIZONS queries and the mapping above; segmentation (each segment's primary and its SOI-crossing `begin`/`end`/`sample` window) is hand-maintained in the script's `CRAFT` dict. Run one craft with `python tools/horizons_trajectory.py <craft> --write` (`<craft>` ∈ `voyager_1`, `voyager_2`, `pioneer_10`); drop `--write` for a dry run that only prints the rows. **Default:** natural osculating cruises + `fix_gaps`, so the engine closes the joins at runtime — HORIZONS queries only, no sim launch. **`--pre-fix`:** Lambert-fits the cruises offline against the running sim's planet positions, so shipped data is gap-free with no runtime fix (slower — launches the sim). `tools/verify_trajectory.py --launch` drives the sim to check each segment's primary and the join continuity. The writer maps row values to columns by header **name** (as the GDScript reader does), reading each table's column order from its header at run time — so a column reorder or insertion in a `.tsv` needs no change here, and a column it writes that's renamed or removed aborts with a message naming it rather than silently misaligning the row.

## Time base & boundary precision

Internal sim time is **TT, J2000 epoch, in seconds** (`program/timekeeper.gd`; JD 2451545.0 = t₀). HORIZONS emits JDTDB; TT ≈ TDB to < 2 ms, so ignore the distinction. `segment_begin`, `segment_end`, and `time_periapsis` are sim seconds: `(JD − 2451545.0) × 86400`. The `epoch_jd` column instead holds a raw JD (the builder converts it). Simplest convention: leave `epoch_jd` blank (= J2000) and write `time_periapsis` as absolute J2000 seconds.

Segment boundary times come from the per-segment `begin`/`end` strings in the converter's `CRAFT` config, via `date_to_jd`. A bare `YYYY-MM-DD` pins the boundary to **midnight** (whole-day granularity); pass `YYYY-MM-DDTHH:MM:SS` for sub-day precision. Day granularity is invisible across a slow heliocentric join — adjacent segments share the *same* boundary value and `fix_gaps` re-fits the cruise to meet its neighbor at that shared instant, so the join closes regardless of the boundary's absolute accuracy — but it misplaces an **open terminal** (the departure start, which is why that begin is pinned to perigee) and any join in a **fast planet-moon frame**, where half a day is hundreds of thousands of km. Give those boundaries an explicit time.

**After editing any `.tsv`, refresh imports** (`<godot-console> --path . --import --headless`) before launching — headless runs use cached table data and silently ignore un-reimported edits. Godot keys reimport on the source's md5, so a regenerate that reproduces identical bytes is correctly a no-op (mtime is not consulted). Note `--editor --headless --quit` quits *before* the async import scan completes — use `--import`, which waits for imports then exits.

## Known imprecisions

The pipeline targets *visual* fidelity (patched conics tuned in-sim), not navigation-grade accuracy. Residuals worth knowing, roughly largest first:

- **HORIZONS solution fidelity** — Pioneer 10 is DE118-era (see *Targets*); the Voyagers are 2022 DE440 refits. This dominates absolute error.
- **TT vs UTC (~50 s in the 1977–89 launch/flyby era, ~69 s today; uncorrected)** — `date_to_jd` treats config calendar dates as TT with uniform days (no leap-second table). A real UTC encounter time transcribed verbatim lands that far off. Because boundaries are synthetic SOI crossings tuned in-sim, and `time_periapsis` comes straight from HORIZONS (already TDB ≈ TT), this shifts absolute *labeling*, not the join geometry — but don't expect a transcribed UTC instant to be second-exact.
- **Default day-granularity boundaries** — see *Time base & boundary precision*: self-consistent at interior joins, but matters at open terminals and in fast frames. Mitigated with an explicit `THH:MM:SS`.
- **Float formatting** — rows print with `%.12g`; at sim-time magnitudes (~1e9 s) that keeps ≳10 ms, and times routed through the JD intermediate carry ~5×10⁻⁵ s — both far finer than 1 s. The cap to be aware of, not a practical limit.
- **Departure perigee is synthetic** — the departure `segment_begin` is the osculating hyperbola's back-projected `time_periapsis`, close to but not exactly liftoff (by design — see the departure-segment note).
